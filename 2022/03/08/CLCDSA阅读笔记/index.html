CLCDSA阅读笔记

Created2022-03-08Updated2022-03-10

背景知识

Cross Language Code Clone （跨语言代码克隆）

  • 跨语言，即跨编程语言

  • 代码克隆，也称作重复代码或者相似代码，通俗的来讲，就是将别处具有当前所需功能的代码挪过来使用。我们平时在自己编写的程序中借鉴他人的代码就类似于这一过程。参见代码克隆检测研究进展.

    代码克隆(code clone), 也称为重复代码(duplicate code)或相似代码, 指的是存在于代码库中两个及以上的相同或者相似的源代码片段.代码克隆产生的原因有很多, 主要是开发者为了提高效率而使用的复用技术, 包括复制粘贴已有的代码片段并进行修改、使用开发框架、复用设计模式等.大量的实证研究表明, 代码克隆广泛存在于各个开源与闭源代码仓库之中, 并且占据了相当比例, 例如, 有研究在Linux系统中检测到了22.3%的代码克隆, Kamiya等人发现在JDK中存在29%的代码克隆, 在某些软件系统中代码克隆甚至达到了50%.广泛存在的代码克隆一定程度上帮助了软件系统的开发, 能够产生正面的效益, 比如可以利用克隆系统测试新增功能对原系统的影响, 然而也有许多研究指出数量巨大的代码克隆会对软件系统造成负面的影响:随着软件生命周期的进行, 未得到良好克隆管理的软件系统会因为代码克隆造成代码库的不断膨胀, 从而增加维护成本.软件缺陷也会因为代码克隆而在系统中被传播, 降低了软件系统的可靠性.所以, 如果不及时控制代码克隆的增长, 对系统的管理、维护、修复等行为都会耗费额外的人力, 导致软件维护成本的提高.

Abstract

  1. 代码克隆对software maintenance and evolution有害。
  2. 现存工具只能针对单一编程语言进行代码克隆检测。
  3. 有些程序可能由于移植性、不同平台的性能表现等原因对同一程序提供不同语言版本的克隆。
  4. 本文提供的CLCDSA通过分析不同编程语言之间的语法特征来检测代码克隆，为了提供对大规模克隆检测的支持，CLCDSA使用跨语言API（编程接口）调用的相似性来过滤非潜在克隆的样本。
  5. CLCDSA使用基于特征向量学习模型的DNN来学习特征和检测克隆，且不需要中间状态。
  6. 关键字 代码克隆 API文档 Word2Vector(词向量化) Source Code Syntax

Introduction

与现存工具的对比

单一语言代码克隆检测工具

CCFinder、Deckard、NiCad、SoucerCC

  • 这些工具仅支持同一编程语言下的代码克隆检测

多语言代码克隆检测工具

CLCMiner

只能检测不同编程语言环境下API名称或调用模式比较相似的代码克隆，对使用不同函数实现相同功能的代码克隆无能为力，这导致在实际的应用场景中CLCMiner很难应用。

LICCA

基于Budimac et al.'s SSQSA architecture提供的中间状态，SQSSA基于源代码提供增强具体语法树（eCST）。LICCA分析eCST并从中生成矩阵，之后将矩阵扩展到代码克隆检测。

然而LICCA具有诸多的限制：

  1. 要求源代码长度相同。

  2. 要求代码执行控制流相同。

这也导致LICCA在实际的应用场景中很难应用。

CLCDSA propose

  1. 不需要中间表示（analyzing source codes without requirement of intermediate representations or restrictions）

  2. 通过9个语法特征来检测代码克隆（detect cross language clones by analyzing the similarity of 9 syntactic source code features）

  3. 动态检测代码不需要预置知识（detect clone on the fly without having any predefined knowledge）

  4. 基于API调用相似性的过滤器（this action filter is designed based on API call similarity used in two different code fragments across different progeamming languages)

Section II CLCDSA designation

A. Feature Selection

Features chosen to work for detecting code clone

  • 声明的变量个数（Number of Variables Declared）

  • 运算符的总数(Total Number of Operators)

  • 参数的个数(Number of Argument)

  • 表达式的个数(Number of Expressions)

  • 运算对象的总数(Total Number of Operands)

  • 循环的个数(Number of Loops(for,while))

  • 抛出的异常个数(Number of Exception Thrown)

  • 引用的异常个数(Number of Exceptions Referenced)

  • McCabes环复杂度(McCabes Cyclomatic Complexity)

    McCabes环复杂度是软件工程中用于衡量软件复杂度的一个指标，它通过测量程序中线性无关路径的数量来计算，可参见McCabe’s Cyclomatic Complexity

B. Preprocessing

Proprocessing consists of four steps.

First. 清除源代码中所有注释、字符串常量

Second. 根据代码段生成抽象语法树(AST)，工具为AntlrV4，这个工具在多平台多语言的条件下仍具有较高的适应性。

Third. 使用正则表达式从AST中分别提取前述9个特征的计量值。

Fourth. 通过NODEVISITOR类的帮助遍历AST来提取API调用。然后将这些API调用传递给下一步(Action Filter)，以过滤掉可能的非克隆的源代码对，从而提高CLCDSA的可伸缩性。
